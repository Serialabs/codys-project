# Data Model – Post‑Purchase Survey App

The following section defines the core database schema for the survey app.  We choose **PostgreSQL** with Prisma (or equivalent ORM) for relational integrity and scalability.  Tables are defined in a conventional SQL style with primary keys (`PK`), foreign keys (`FK`), indexes and relationships.

## Entities and relationships

### `shops`
Stores metadata about each Shopify store that has installed the app.

| Field | Type | Notes |
| --- | --- | --- |
| `id` | UUID (PK) | Internal identifier. |
| `shop_domain` | VARCHAR UNIQUE | The myshopify.com domain; used to identify the shop. |
| `access_token` | TEXT | OAuth access token for API calls. |
| `plan` | ENUM (`starter`, `analyst`, `pro`) | Current subscription plan. |
| `installed_at` | TIMESTAMP | When the app was installed. |
| `uninstalled_at` | TIMESTAMP nullable | When the app was uninstalled (soft deletion). |

### `users`
Represents admin users of the app (if multi‑user support is added later).

| Field | Type | Notes |
| --- | --- | --- |
| `id` | UUID (PK) | User ID. |
| `shop_id` | UUID (FK → shops.id) | Belongs to a shop. |
| `email` | VARCHAR | User’s email. |
| `role` | ENUM (`owner`, `editor`, `viewer`) | Permission level. |
| `created_at` | TIMESTAMP | Timestamp of user creation. |

### `surveys`
Top‑level container for a set of questions and actions.

| Field | Type | Notes |
| --- | --- | --- |
| `id` | UUID (PK) | Survey ID. |
| `shop_id` | UUID (FK → shops.id) | Owner shop. |
| `title` | VARCHAR | Display name. |
| `status` | ENUM (`draft`, `published`, `archived`) | Publication state. |
| `created_at` | TIMESTAMP | When created. |
| `updated_at` | TIMESTAMP | Updated whenever questions change. |
| `promo_action_id` | UUID nullable (FK → promo_actions.id) | Optional action to show on completion. |

### `questions`
Defines each question in a survey.

| Field | Type | Notes |
| --- | --- | --- |
| `id` | UUID (PK) | Question ID. |
| `survey_id` | UUID (FK → surveys.id) | Parent survey. |
| `type` | ENUM (`radio`, `checkbox`, `text_short`, `text_long`) | Basic question types for MVP. |
| `prompt` | TEXT | The question text. |
| `is_required` | BOOLEAN | Whether the question must be answered. |
| `position` | INTEGER | Sort order in the survey. |
| `follow_up_parent_id` | UUID nullable (FK → questions.id) | If this is a follow‑up, reference to the parent question. |
| `follow_up_value` | VARCHAR nullable | Value that triggers this follow‑up (for radio/checkbox questions). |

### `choices`
Stores the possible options for radio and checkbox questions.

| Field | Type | Notes |
| --- | --- | --- |
| `id` | UUID (PK) | Choice ID. |
| `question_id` | UUID (FK → questions.id) | Parent question. |
| `label` | VARCHAR | Display text. |
| `value` | VARCHAR | Machine‑readable identifier (slug). |
| `position` | INTEGER | Sort order. |

### `audiences` (Target rules)
Defines criteria for who should see a survey.

| Field | Type | Notes |
| --- | --- | --- |
| `id` | UUID (PK) | Audience ID. |
| `survey_id` | UUID (FK → surveys.id) | Survey this audience belongs to. |
| `attribute` | ENUM (`lifetime_orders`, `product_id`) | For MVP, support new vs returning customers and product targeting. |
| `operator` | ENUM (`eq`, `gt`, `lt`, `in`) | Comparison operator. |
| `value` | VARCHAR | Comparison value (e.g., `1` or `1001,1002`). |
| `created_at` | TIMESTAMP | Timestamp. |

### `responses`
Captures each survey submission.

| Field | Type | Notes |
| --- | --- | --- |
| `id` | UUID (PK) | Response ID. |
| `survey_id` | UUID (FK → surveys.id) | Which survey. |
| `shop_id` | UUID (FK → shops.id) | Which shop. |
| `order_id` | VARCHAR | Shopify order ID; stored as string for flexibility. |
| `customer_id` | VARCHAR nullable | Shopify customer ID if available. |
| `audience_id` | UUID nullable (FK → audiences.id) | Audience matched. |
| `completed_at` | TIMESTAMP | When the survey was completed. |
| `promo_code_id` | UUID nullable (FK → promo_codes.id) | Discount code used in action. |

### `response_items`
Stores each answer within a response.

| Field | Type | Notes |
| --- | --- | --- |
| `id` | UUID (PK) | Item ID. |
| `response_id` | UUID (FK → responses.id) | Parent response. |
| `question_id` | UUID (FK → questions.id) | Which question was answered. |
| `choice_value` | VARCHAR nullable | Selected value(s) for radio/checkbox questions; multiple values can be comma‑separated. |
| `text_answer` | TEXT nullable | Response for text questions. |

### `promo_codes`
Tracks discount codes generated by actions.

| Field | Type | Notes |
| --- | --- | --- |
| `id` | UUID (PK) | Promo code ID. |
| `shop_id` | UUID (FK → shops.id) | Associated shop. |
| `survey_id` | UUID (FK → surveys.id) | Survey that issued this code. |
| `code` | VARCHAR | The discount code string. |
| `created_at` | TIMESTAMP | When generated. |
| `expires_at` | TIMESTAMP nullable | Expiration date. |
| `is_redeemed` | BOOLEAN | Whether the code has been redeemed. |

### `integrations`
Config table storing external service credentials and settings per shop (e.g., Klaviyo API key, Flow settings).

| Field | Type | Notes |
| --- | --- | --- |
| `id` | UUID (PK) | Integration ID. |
| `shop_id` | UUID (FK → shops.id) | Owner shop. |
| `type` | ENUM (`klaviyo`, `flow`, `webhook`) | Integration type. |
| `config_json` | JSONB | Arbitrary configuration (e.g., API keys). |
| `enabled` | BOOLEAN | Whether integration is active. |

## Indexing strategy

* **shops.shop_domain** – Unique index for quick lookup when validating signed requests.  
* **surveys.shop_id**, **questions.survey_id**, **responses.survey_id** – Foreign key indexes to speed join queries.  
* **responses.order_id** – Index to quickly join responses with orders in revenue analysis.  
* **audiences.survey_id** – To find rules for a given survey.  
* **response_items.question_id** – To aggregate answers per question.  
* **promo_codes.code** – Unique index to ensure no duplicate codes.

## Data retention policy

* **Survey responses** – Retained indefinitely by default to allow longitudinal analysis and benchmarks.  Merchants can request deletion of individual responses to comply with GDPR/CCPA; responses older than three years may be purged on request.  
* **Promo codes** – Retained until one year after expiration to audit redemptions.  
* **Access tokens** – Rotated if the merchant reinstall or changes permissions; stored encrypted at rest.  
* **Soft deletes** – Surveys and audiences are soft‑deleted (archived) so historical responses remain accessible even if the survey is removed.

## Notes

* For advanced question types in V2 (dropdown, slider, etc.), add additional enum values in `questions.type` and consider normalising slider ranges or ranking options.  
* Multi‑survey rotation will require a `survey_groups` table to group surveys and store rotation logic (mutually exclusive, sequential, round robin).  
* To support translations (V2), create `question_translations` table keyed by question and locale.
